# Boya 麻将2 HAR 本地还原与可控服务经验手册

> 来源任务：`019f452e-9764-7ca0-89ef-745eafe70baf`
> 整理日期：2026-07-10
> 适用范围：HAR 还原 Cocos 客户端、本地 protobuf-over-WebSocket 服务、确定性测试场景、概率模式、本地用户、SQLite 历史和真实浏览器验收。
> 状态说明：来源任务在本文整理时仍在做最后一轮逐图复核。本文把已反复验证的经验与最新修复分开记录；最终发布结论仍应以当前代码重新跑出的报告为准。

## 1. 最重要的结论

1. **HAR 回放只是起点，不是“本地可控服务”的终点。** 完整本地化至少包含本地静态资源、本地 WebSocket、独立心跳、本地规则引擎、SQLite、用户归属、管理后台和外部请求审计。
2. **协议自洽不等于客户端视觉正确。** `mismatches=0`、金额公式正确、单测通过，仍可能出现错误高亮、错误消除、无限旋转或动画卡死。
3. **每个游戏帧必须作为完整状态处理。** 盘面、`lines`、金额、倍率、状态、上下补牌、金牌转 Wild、免费次数和余额必须一致，不能只改其中一个字段。
4. **HAR 认证身份和本地用户身份必须分层。** Cocos 内部继续使用兼容 HAR 的认证 token；`user1`、`user2` 等只作为本地服务用户键，不能直接替换客户端认证串。
5. **真实浏览器逐场景验收不可省略。** 必须从真实按钮进入、等待每次动画和终局、检查网络与 WebSocket，再人工看关键截图。只调用内部 JS 或只看 JSON 报告都不够。
6. **任何代码变化后都要重跑最终报告。** 旧 PASS 报告只能证明旧代码，不能作为新盘面、新协议字段或新截图逻辑的证据。

## 2. 项目最终形成的分层

```text
本地入口壳 /__game/test、/__game/live?token=user1
    -> 本地 Cocos 静态资源 local-har-client/boya-mahjong2/
    -> 本地大厅与游戏 WebSocket /gate/ws
    -> HAR 兼容层 tools/lib/boya-har.mjs
    -> 本地 responder tools/local/server/
    -> 规则引擎 tools/local/engine/
    -> SQLite local-data/boya-mahjong2.sqlite3
    -> 后台 /__admin、历史 /__history、健康检查 /api/health
```

各层职责必须分开：

- HAR 层负责保留客户端能接受的 URL、认证形态和 protobuf 外壳。
- 规则引擎只负责盘面、Ways、消除、重力、补牌、Wild、倍率和金额，不感知 wire 字节。
- responder 把引擎结果映射到 `40001/40003/40005/40007` 等响应。
- SQLite 是用户余额、配置版本、局、步骤、历史和 RTP 的持久化真值。
- Playwright 负责验证真实 Cocos 状态机和视觉表现，不代替规则单测。

## 3. 从对话中沉淀出的协议真值

| 请求 -> 响应 | 用途 | 经验 |
|---|---|---|
| `40000 -> 40001` | 进入游戏 | 初始余额必须直接来自本地用户，不能先显示 HAR 余额再跳变。 |
| `40002 -> 40003` | 普通旋转 | 可以承载真实普通中奖，但必须同时提供正确盘面、`lines`、金额和合法终止状态。 |
| `40006 -> 40007` | 购买/触发免费 | 3/4/5/6 个胡分别进入 10/12/14/15 次免费流程。 |
| `40004 -> 40005` | 免费旋转及级联 | 同一免费旋转内部的级联不能重复扣免费次数；下注档位必须传到所有免费帧。 |
| `5000 -> 5001` | 心跳 | 必须独立生成响应，绝不能消耗 HAR 顺序回放游标。 |
| `20047/20048`、`20051/20052` | 游戏内历史列表/详情 | 必须按当前本地 token 隔离，跨用户详情返回 404。 |

关键一致性公式：

```text
roundWin == sum(lines[].score)
totalWin == 当前局按协议定义的累计金额
line 的 iconId == 客户端实际高亮符号
line 的 Ways == 从第 1 轴起连续出现的基础符号、对应金牌或 Wild 的组合数
结算余额 == 入场余额 - 普通下注 - 购买费用 + 总中奖
RTP == 总中奖 / (普通下注 + 购买费用)
```

免费旋转不再次计入 RTP 分母，购买费用需要计入。

## 4. 结论演进：哪些早期判断后来被修正

### 4.1 “基础 `40003` 不能中奖”不是最终结论

早期把一个输盘面的 `40003` 只改成中奖金额，或强塞不完整 `lines`，会导致错误动画、自动重转和认证超时，因此一度改用 `40006/40007/40004/40005` 免费链路承载中奖。

后续真实浏览器验证证明，问题不是 `40003` 天生不能中奖，而是**单个伪造中奖帧没有满足完整状态机**。当前普通小奖模式可以走 `40002 -> 40003`，但中奖帧之后还要提供合法结算/终止帧，并保证盘面、路线、金额和补牌一致。

因此不能把阶段性绕路写成协议禁令。判断某命令是否可用，必须基于完整请求序列和客户端最终状态。

### 4.2 `mismatches=0` 不能证明视觉正确

来源任务曾在协议矩阵通过后，仍由用户截图发现错误消除。根因是 `topResult` 的帧归属放反：

- 正确语义：当前中奖帧的 `topResult` 是“本次消除后即将掉入的牌”。
- 错误实现：把这些牌写进下一帧。
- 结果：客户端先用旧补牌组成一个盘面，再按下一帧 `lines` 高亮另一种符号，表现为消除元素错位，并可能引发长转。

修复门禁应直接断言：每个中奖帧的 `topResult` 必须等于该帧消除后进入下一盘的实际补牌。

### 4.3 固定截图延迟不能代表固定动画阶段

早期脚本统一在响应后约 2.7 秒截图。多步级联时，后续步骤的 2.7 秒可能已经进入消除或补牌，文件虽然叫 `highlight.png`，内容却不是高亮帧。

正确做法：

1. 首次遇到新动画时以 250ms 间隔抓时间线。
2. 找出每一步真实高亮窗口。
3. 正式矩阵为每个中奖步骤保存早/中/晚三帧，不再把单一时间点当成真值。
4. 首段与后续级联分别设置等待条件或时间组；同一组内再选择实际命中高亮的帧作为正式证据。
5. 同时保存协议步骤，不以截图文件名推断动画阶段。

### 4.4 场景名称和数学合法都不等于业务场景正确

两个真实例子：

- 普通路线场景误放 3 个胡，数学仍可能自洽，但客户端会进入特殊胡牌/免费流程并长时间不发送下一次级联请求。
- Wild 场景的填充符号额外形成一条合法中奖线，导致声明“一条路线”却实际出现多条高亮。

场景必须带精确元数据并由引擎结果反验：中奖步骤数、终止步骤、倍率、胡数量、Wild 数量、金转 Wild、精确或允许的 line 集合。普通旋转任一步最多 2 个胡；3 个及以上必须进入购买/免费状态。

## 5. 本地 token 的正确合同

外部地址可以保持可读：

```text
http://127.0.0.1:18082/__game/live?token=user1
http://127.0.0.1:18082/__game/live?token=user2
http://127.0.0.1:18082/__game/live?token=usergame1
```

但内部必须分成两个身份：

```text
地址栏 / 本地用户键: user1
Cocos HTTP 认证 token: <HAR_AUTH_TOKEN>
本地 WebSocket 参数: userToken=user1
```

踩坑与修复：

- 直接把 Cocos HTTP token 换成 `user1`，客户端会在建立 WebSocket 前停止。
- Cocos 建连时可能在 `userToken` 后追加缓存后缀；网关只剥离该客户端追加部分，再按严格规则校验原 token。
- 本地入口使用同源全屏壳包住内部 Cocos 页面，地址栏始终保留短 token，内部仍保持 HAR 兼容。
- session、余额、历史、RTP 和历史详情所有权都绑定本地用户，不能使用 HAR token 作为数据库主键。
- token 不存在时本地创建；非法 token 返回 400，不能静默映射到其他用户。

## 6. “完全本地”必须如何证明

不能只因为入口是 `127.0.0.1` 就宣布完全本地。最低证明链：

1. Cocos JS、JSON、图片、音频全部由当前服务返回。
2. 大厅和游戏两条 WebSocket 都指向当前 loopback 服务。
3. `gateConfig` 等默认配置不能保留线上网关兜底；缺少 URL 参数时也不能回落外网。
4. 测试客户端、用户客户端、后台和历史页分别记录全部 HTTP 与 WebSocket URL。
5. 只允许当前 `127.0.0.1:18082`、`data:` 和 `blob:`；外部主机、请求失败、HTTP 4xx/5xx 均为 0。
6. 规则、概率、用户、余额和历史都来自当前仓库与本地 SQLite，不依赖生产账号、钱包、网关或 `slot-platform` 运行态。

对应审计脚本：

```bash
node tests/playwright/boya-mahjong2-local-runtime-audit.mjs
```

## 7. 高频故障与排查顺序

| 现象 | 常见根因 | 优先检查 | 正确修复方向 |
|---|---|---|---|
| 图片副本打不开 | Cocos 资源头混淆或扩展名与真实格式不一致 | `file image/*/*`、源资源头、manifest | 只修复预览副本；运行时目录保持原路径和原字节。 |
| 进入或中奖时 404 | HAR 未捕获懒加载 import/native/动画资源 | 浏览器 network、控制台 UUID、Cocos 配置引用 | 补到原运行路径并记入 `recovered-assets.json`，重新跑真实动画。 |
| 多次下注后认证超时 | `5000` 心跳消耗 replay 游标，样本耗尽 | `5000/5001` 数量、WS close、历史 source | 所有模式独立生成 `5001`，长测至少 70 秒。 |
| 一换成短 token 就不连接 | 把本地用户 token 当成 Cocos 认证 token | 页面是否创建两条 WS、进入帧是否出现 | 分离 HAR 认证和本地 `userToken`。 |
| 用户 token 多出随机后缀 | Cocos 在 WS URL 后追加缓存串 | 服务端原始升级 URL | 只在网关层剥离已知追加部分，之后再严格校验。 |
| 分数对但盘面不亮 | 输盘面加假金额、错误 `iconId`、缺失 `lines` | 解码盘面、Ways、line、截图 | 由统一引擎计算盘面和路线，禁止只改金额。 |
| 高亮与消除的牌不同 | `topResult` 写到下一帧或截图时机错 | 250ms 时间线、相邻步骤补牌 | 把补牌归给触发掉落的当前中奖帧，并重新定截图窗口。 |
| 一直旋转/自动按钮转圈 | 没有合法零中奖终局、场景误触发 3 胡、级联状态错位 | 终局 `40003/40005`、额外 `40002`、WS 时序 | 每局必须回到可再次点击状态；终局后不得无点击产生新旋转。 |
| 下注档位看似没生效 | 只改首个购买帧，免费帧仍继承 HAR 旧值 | `betMulti`、`betCoin`、购买价、免费详情 | 当前档位贯穿购买响应、全部免费帧、奖金和历史。 |
| 历史串用户 | session 未绑定用户或详情缺所有权检查 | user/session/round 关联、跨用户详情 | SQLite 外键式归属；列表和详情都按 token 限制。 |
| 引擎算出字段但客户端收不到 | protobuf 重写器只替换原帧已有 packed 字段 | 原始帧是否存在目标 field | 允许追加缺失的合法 override 字段，并加解码回归测试。 |

## 8. 推荐的实现顺序

以后复用到其他 HAR 游戏时，按下面顺序推进，避免过早伪造完整规则：

1. **仓库与 HAR 盘点**：先看 dirty workspace、HAR 类型、HTTP 资源、两条 WS、真实命令序列和连接归属。
2. **写协议事实表**：记录请求/响应、字段、状态转换、心跳和历史，未知字段明确标注未知。
3. **完成最小 replay**：本地静态资源、本地 WS、进入、真实 UI bet、历史日志。
4. **先独立心跳**：在扩展数据集之前处理 `5000/5001`，并做长时间浏览器验证。
5. **补齐资源审计**：从真实浏览器发现 HAR 漏项；预览副本和运行时资源分开管理。
6. **增加 dataset**：先循环真实 HAR 原子帧，不在协议未知时硬造中奖。
7. **抽离规则引擎**：盘面、Ways、金额、级联、Wild 和 validator 与 protobuf 分层。
8. **建立确定性场景**：场景只定义初始盘面/seed/补牌队列；`lines` 和奖金由引擎计算。
9. **接概率、SQLite 和后台**：每局固定配置快照；余额和历史在事务内更新。
10. **接本地用户**：外部短 token、内部 HAR 认证、WS 用户参数三者分层。
11. **逐场景真机验收**：真实点击、协议解码、终局、网络、截图和人工复核一起做。
12. **最后做完全本地审计**：检查正常入口和所有兜底路径都不会访问外部主机。

## 9. 浏览器验收的最低门禁

### 9.1 每个确定性场景

- 真实浏览器进入并点击，不只调用 `GameNet.requestNormalSpin()`。
- 每个中奖步骤保存解码后的盘面、`lines`、金额、倍率、胡/Wild/金牌位置和补牌。
- 等待零中奖终局，确认按钮恢复可再次点击。
- 终局后额外观察，未点击时不得出现新的 `40002`。
- HTTP 错误、页面错误、WS 非预期 close、认证超时和服务端 mismatch 全为 0。
- 对关键场景查看正确时间窗口的高亮截图，至少覆盖多步级联、Wild、金转 Wild、Scatter 重力和级联上限。

### 9.2 live 概率模式

- 连续至少 10 局，每局都有合法终局，不能偶发长转。
- 验证自动旋转、下注切换、购买免费和免费旋转。
- 验证客户端历史列表和详情来自 SQLite。
- 验证 `user1/user2/usergame1` 余额、历史和 RTP 相互隔离。
- 后台桌面和手机布局都能看清用户、概率与历史主操作。

### 9.3 视觉复核

自动报告之后仍要人工看：

- 高亮符号是否等于 `lines.iconId`。
- 高亮路径是否从第 1 轴连续。
- 消除后的幸存牌顺序、金转 Wild 和新补牌是否与下一盘一致。
- 显示金额是否等于协议金额。
- 截图是否真的处于高亮阶段，而不是旋转、消除后或静止盘面。

## 10. 明确禁止的“假通过”

- 只修改 `roundWin/totalWin`，不改盘面和 `lines`。
- 修改历史日志金额冒充客户端真实中奖。
- 只用 JS 直接发 spin，未验证真实按钮和 Cocos 状态机。
- 只看 `mismatches=0`、单测或 REST API，不看浏览器动画。
- 用文件名 `highlight.png` 代替确认截图内容。
- 普通场景误放 3 个以上胡，却继续按普通旋转处理。
- 场景标签写“一个 Wild”，但不从真实生成结果验证数量和路线。
- 正常入口本地化，但保留线上默认网关或资源兜底。
- 改代码后继续引用改动前的 PASS 报告。
- 为了让测试过而放宽预期线路数量、忽略 404 或忽略 WS close。

## 12. 证据与维护规则

稳定报告、最终截图和失败探针要分开：

- 最终报告只保存当前代码的 fresh PASS。
- 失败轮次保留在本地 probe 目录用于定位，但不要混入最终交付。
- 报告必须记录代码时间、服务端口、场景数、命令计数、错误计数和输出目录。
- 盘面或协议逻辑变化后，至少重跑受影响的定点场景和完整矩阵。
- 本文记录方法论；字段真值或当前实现发生变化时，优先更新对应 Spec，再同步修订本文中过时结论。
